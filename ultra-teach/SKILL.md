# Ultra Teach — Project Onboarding

Run **once per project**. Interview the user, write `.ultra.md` to the project root, and validate or scaffold `_shared/*.md` in the skill pack. All Apple Ultra review and design skills read `.ultra.md` automatically when present.

**Downstream consumers:** `apple-design-head`, `ultra-visual-system`, `ultra-component-discovery`, `ultra-brand-voice`, `ultra-pr-ship-review`, executive skills.

---

## When to use

- First install of Apple Ultra Skills in a repo
- User says "onboard", "teach", "set up skills", "fill shared context"
- Reviews feel generic — `.ultra.md` missing or stale
- After major pivot (new product type, rebrand, new market)

---

## Interview protocol

Ask **one section at a time**. Do not rush — good context makes every downstream skill more accurate.

### Section 1 — Product overview

- What is this product in one sentence (customer-facing)?
- Stage: pre-launch · early · growth · mature
- Primary user action in one verb phrase

### Section 2 — Register & product type

**Register** (shapes design evaluation):

| Register | Goal |
|----------|------|
| **Brand** | Distinctiveness — design IS the product (marketing, campaigns) |
| **Product** | Familiarity — design SERVES the task (app, dashboard, booking) |

**Product type:** B2B SaaS · consumer · e-commerce · internal tool · AI-native · scheduling/booking · other

### Section 3 — Users

- Primary user role and technical sophistication
- Job-to-be-done (what they hired the product for)
- Failure mode (what happens if the product lets them down)

### Section 4 — Design system

- Existing design system or component library name
- Key tokens: colors, type, spacing constraints
- **Named rules** (format: `**The [Name] Rule.** [one sentence doctrine]`) — minimum 2, max 6
- **Creative North Star** — named metaphor (e.g. "The Lab Notebook", "The Corner Bookshop")
- Off-limits components or areas

### Section 5 — Benchmarks

- Positive references (products to aspire to)
- Anti-references (products to avoid resembling)
- Priority craft dimension: clarity · responsiveness · intelligence · seamlessness · power

### Section 6 — Stack & paths (engineering)

Confirm or fill `_shared/STACK.md` and `_shared/PATHS.md`:
- Verify command (`npm run verify` or equivalent)
- Migrations directory and schema file paths
- API auth patterns (cron, dashboard, public API prefixes)

### Section 7 — Known weaknesses

- Biggest known UX or engineering problem
- Broken or incomplete flows
- Off-limits areas for audits

## Output: `.ultra.md`

Write to **project root**:

```markdown

*Generated by ultra-teach. Re-run ultra-teach to update.*

## Product
[One sentence]
Stage: [pre-launch | early | growth | mature]
Primary action: [verb phrase]
Type: [B2B SaaS | consumer | e-commerce | internal | AI-native | scheduling | other]
Register: [Brand | Product]

## Users
Primary user: [role + sophistication]
Job-to-be-done: [what they hire the product for]
Failure mode: [consequence of product failure]

## Design
Creative North Star: [named metaphor]
System: [design system name or "custom"]
Key tokens: [colors, type, spacing summary]

Named Rules:
- **The [Name] Rule.** [doctrine]
- **The [Name] Rule.** [doctrine]

## Benchmarks
Positive: [list]
Anti: [list]
Priority dimension: [clarity | responsiveness | intelligence | seamlessness | power]

## Stack
Verify: [command]
Migrations: [path]
Schema: [path]
API patterns: [see _shared/PATHS.md or inline summary]

## Known Issues

Off-limits: [areas]
```
